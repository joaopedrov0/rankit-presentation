M7 - SOLID (O) ✅

Talvez nenhum princípio do SOLID se destaque tanto quanto o Open-Closed Principle no nosso projeto. A proposta do projeto está intrínsecamente associada ao conceito de facilitar a expansão futura do projeto.

Logo abaixo temos um método da classe Database que faz uma busca por uma obra específica e retorna ela de modo estruturado para que possa ser devidamente renderizado na página da obra.

Note que eu disse explicitamente que os dados voltam de forma estruturada, isso significa que não importa o tipo de mídia que for inserido aqui, ela deve retornar sempre no mesmo "padrão".

Pra que?

Para responder essa pergunta eu vou contar sobre um acontecimento que aconteceu durante a implementação de uma das funcionalidades da aplicação, a integração com a API de jogos.

Quando a integração com a API de jogos foi feita, na hora de fazer a conexão disso com o método do banco que retorna os dados de modo padronizado, foi criada a classe para estruturar os dados que viriam dessa nova API. Depois de feita essa padronização dos dados, a página da obra foi renderizada corretamente. Não foi preciso alterar uma linha sequer do restante do código para fazer com que a aplicação aceitasse jogos nas avaliações.

M8 e M9 - Injeção de dependência e Associação bilateral ❓

Injeção de dependência é um conceito indispensável para esse projeto, porém o uso dele a nível de código em uma injeção literal de dependência ou associação bilateral entre classes/objetos não fazia sentido.

Mas por quê?

Como estamos trabalhando com banco de dados, apesar de criarmos uma classe "usuário" (User) e uma classe "review" (Review), que tem essa relação de existência dependente entre si (uma review não existe sem um usuário), quando a review é criada o usuário já está no banco de dados. Isso implica que ele já foi convertido em um dicionário para ser armazenado no banco, ou seja, ele não existe como uma instância de classe.

Também ocorre algo similar com a associação bilateral. Da forma que foi organizado os dados, um usuário do banco sabe as obras que ele viu, da mesma forma que a mesma obra sabe quem viu ela.

M14 e M15 - Interfaces e SOLID (I) ✅

Foi criado um contrato (interface) para a classe Database que usaríamos no projeto, facilitando o processo de uma futura troca de banco de dados se fosse necessário.

SOLID (I) (Interface Segregation Principle) foi usado para realizar contratos menos genéricos com a classe Database, dividindo quais seriam as responsabilidades e métodos de cada contrato.

Depois de criadas as interfaces, elas foram usadas com o uso do método .register()

M16 - SOLID (D) ✅

Para demonstrar o princípio da Inversão de Dependência do SOLID, escolhemos mais uma vez a classe Database, porém dessa vez revelando um detalhe de implementação que estava oculto nos outros exemplos da mesma classe, que é o uso da classe abstrata como tipo esperado a ser recebido do método de inserção no banco de dados.

M17 - Agregação de classes ✅

Para demonstrar o conceito da agregação de classes, escolhemos a classe LoginManager, que tem a responsabilidade de validar logins e armazenar os logins ativos em cache. Esta segunda responsabilidade é feita através de um atributo da instância de LoginManager que é um dicionário cuja chave é um id de usuário e o valor é um objeto da classe LoginCache, que contém os dados dos logins ativos salvos em cache por questões de otimização.

M18 - Composição de classes ❌

Durante o desenvolvimento do projeto não foi usado composição de classes. Porém, nesse conceito específico, isso seria um conceito bem vindo na classe LoginManager. E o motivo disso é que ela tem duas responsabilidades, como eu havia mencionado anteriormente, que é validar as tentativas de login e manusear o cache dos logins ativos. Portanto, seria possível através da composição de classes, tornar a classe LoginManager composta de outras duas classes, onde cada uma tem apenas uma dessas responsabilidades.

Porém, como eu disse, até o atual momento do projeto, não foi implementado dessa forma, por isso o conceito está marcado com um ❌.

Com as funcionalidades atuais do projeto, não vimos necessidade de implementar árvores binárias até então, ao menos não de forma direta. Entretanto, um vislumbre de algo que estudamos para tentar implementar (mas não tivemos sucesso, por isso não está marcado), foi a busca de perfis de usuário que, por não vir de nenhuma API externa, nos permitia cogitar criar nosso próprio método de busca. M7 - Grafos ✅

Os grafos foram utilizados na nova funcionalidade de sugestão de perfis. Ao seguir um perfil, o sistema gera um grafo para mapear os seguidores do perfil visitado e do usuário logado, e após isso ele retorna uma sugestão personalizada de perfis para ele seguir que vai de acordo com perfis similares ao que ele acabou de seguir, mas que ele ainda não segue.
